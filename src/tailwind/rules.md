# Правила написания кода на tailwind

> [!IMPORTANT]
>
> #### Перед началом работы с проектом на tailwind нужно **ознакомиться с документацией**
>
> #### Особое внимание уделить [основам и принципам](https://tailwindcss.com/docs/utility-first), на которых базируется tailwind.
>
> Без прочтения документации и понимания tailwind нет смысла его использовать.

## Не изобретать велосипеды

В Tailwind отличная документация, практически всё, что необходимо для написания кода, уже реализовано, остаётся найти и использовать.

Использование стандартных классов, предоставляемых tailwind позволяет экономить ресурсы как на разработку, так и на сопровождение кода.

Tailwind предоставляет возможность достаточно гибкой настройки и кастомизации, поэтому когда вам кажется, что возникает необходимость в велосипедостроении - стоит подумать о том, что бы превратить велосипед в стандартные классы TW, через конфиг, через npm-пакет или слои.

## Не писать CSS

**В идеале в компоненте не должно существовать блока со стилями**

Каждое использование блока со стилями добавляет лишний CSS код и приводит к усложнению приложения (протекание стилей, беспорядок в нейминге, беспорядок в подходе к написанию стилей), а значит увеличению времени на разработку и сопровождение кода.

Однако, идеалов не существует, поэтому важно чётко понимать необходимость и выгоду от использования блока со стилями.

В каких ситуациях допускается писать стили:

- Когда невозможно использовать стандартные классы TW.
- Когда элемент используется много раз и содержит большое количество классов.
- Когда нужно переопределить стилизацию внешнего компонента.
- Когда технически выгоднее написать css вместо использования классов TW.

Если необходимость написания CSS кода всё же возникла, стоит в первую очередь подумать об использовании `@apply` вместо чистого css.

## @apply - зло

Необходимо избегать использования `@apply` т.к. каждое использование раздувает CSS код.

Однако есть ситуации, когда это зло необходимо:

- Когда нужно переопределить стилизацию внешнего компонента.
- Когда необходимо писать код в слоях.
- Когда технически выгоднее написать `@apply` вместо классов в html коде.

## Нет квадратным скобкам

**Необходимо избегать использования произвольных значений**

Произвольные значения, или arbitrary values, или [квадратные скобки](https://tailwindcss.com/docs/adding-custom-styles#using-arbitrary-values) приводят к двум большим проблемам:

1. Раздувается CSS код т.к. каждое произвольное значение - это дополнительные css строки кода.
2. Усложняется разработка и сопровождение проекта. Каждое произвольное значение - это нестандартный код, который, возможно, является результатом ошибки в дизайне.

Для того что бы избегать произвольного кода, достаточно ответить на следующие вопросы:

- Нет ли ошибки в дизайне?
- Возможно ли реализовать то же самое на стандартных классах?
- Возможно ли вынести произвольное значение в конфигурацию или слои?

Если ответы на все вопросы - отрицательные, только в этом случае стоит использовать произвольные значения.
