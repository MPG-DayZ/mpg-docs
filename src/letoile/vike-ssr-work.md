# Vike SSR

## Для чего SSR

Рендеринг на стороне сервера (сокращенно «SSR») позволяет генерировать HTML из компонентов Vue на сервере и отправлять
этот HTML вашим пользователям. SSR позволяет вашим пользователям видеть содержимое страницы до того, как ваш JavaScript
бандл загрузится и запустится.

SSR во Vue происходит в несколько этапов:

- На сервере получаем данные для всего/ некоторой части приложения.
- Затем на сервере рендерится приложение в HTML и отправляется в ответе.
- Затем на клиенте грузится код JavaScript для приложения.
- Затем на клиенте подключается JavaScript логика к сгенерированному сервером HTML для приложения (это «гидрация»).

Ключевой момент заключается в том, что каждый шаг должен был завершиться для всего приложения одновременно, прежде чем
можно было начать следующий шаг. Это неэффективно, если некоторые части вашего приложения работают медленнее других, как
это происходит практически с каждым нетривиальным приложением.

Vue позволяет через `<Suspense>` разбить ваше приложение на более мелкие независимые блоки, которые будут выполнять эти
шаги независимо друг от друга и которые по идее (но не факт) не будут блокировать остальную часть приложения. В
результате пользователи вашего приложения увидят контент раньше и смогут начать с ним взаимодействовать гораздо быстрее.
Самая медленная часть вашего приложения не будет тянуть за собой быстрые части. Эти улучшения происходят автоматически,
и для их работы не нужно писать специальный код координации.

## Как работать с SSR в разрезе Vike, и что нужно понимть

По дефолту в vike все SSR, но в настройках +config.ts можно отключить SSR для всех страниц или только для некоторых.

Самое важное, что требуется от разработчика при работе с SSR - это понять ряд вещей:

- Нужно как можно быстрее отдать пользователю неинтерактивную (так как HTML с сервера) версию нашего приложения, чтобы
  он мог просматривать статический контент, пока он ожидает загрузки JS.
  То есть если этого не требует SEO и нет ошибок гидрации (дерево, создаваемое нашими компонентами в браузере, должно
  соответствовать дереву, созданному сервером), то отдаем статичный контент без запросов в API (например к профилю и
  тд), тем самым не блокируя общий поток ожиданием получения всех этих специфичных данных на сервере, прежде чем мы
  сможем начать отправку HTML кода клиенту
- Если ситуация другая - нам необходимо собрать все данные на сервере достаточные для отрисовки первого экрана, прежде
  чем отправить HTML- код клиенту, выполняется это в [+data.ts](/letoile/data-fetching), и там же
  минимизируется/обрабатывается и так далее для дальнейшей передачи во Vue компоненты
- Все что не влезает в один экран нужно обернуть в `<Client-only>` из [vike-vue](https://vike.dev/vike-vue), и
  соответственно данные нужные для отрисовки этих обернутых компонентов в `+data.ts` получать не нужно
- Помнить нюанс с гидрацией - Vue гидрирует дерево за один проход. Это означает, что после начала гидрации (что, по
  сути, вызывает функции нашего компонента), Vue не остановится, пока не закончит делать это для всего дерева. Как
  следствие, нам придется дождаться гидрации всех компонентов, прежде чем мы сможем взаимодействовать с любым из них.

## Stream (!стоит проверить)

Так как разработчики грозятся что в vike-vue интегрирован HTML Streaming и при создании сервера на экспрессе мы включили
стрим, то определенно стоит ориентироваться на способ, который очень детально и подробной
в [статье про SSR](https://github.com/reactwg/react-18/discussions/37) описал и наконец расставил все точки Ден Абрамов,
а именно под заголовками `Streaming HTML before all the data is fetched`
и `Hydrating the page before all the code has loaded`
