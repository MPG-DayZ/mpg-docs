# Структура проекта Letoile

Командой было принято решение использовать упорядоченную структуру папок, на основе модулей, которая бы задавала
архитектуру с разными слоями абстракции, и имела однонаправленный поток данных (импортов) между этими слоями.

Пример такой структуры на основании выбранного стека Vue+Vike

```
src
  │
  ├── routes
  │  ├── account
  │  ├── cartCheckout
  │  ├── home
  │  ├── product
  │  └── staticPages
  │
  └── shared
```

- `/routes` - Каталог со страницами либо группой страниц, которая отвечает за определенный бизнес функционал
- `/routes/*` - Группы роутов(модулей) - самостоятельные сущности со своей бизнес-логикой, со своей зоной
  ответственности, со своей задачей, связки компонентов, имеют свои хелперы и апи, чтоб решать свою конкретную задачу
- `/shared` - Содержит компоненты, хелперы, плагины и т.д, которые могут использоваться внутри остальных групп роутов

## Общий слой Shared

```
shared
  │
  ├── components
  │  ├── DocumentsTable
  │  │  ├── __tests__
  │  │  ├── DocumentsTable.vue
  │  │  ├── TableCheckbox.vue
  │  │  └──index.ts
  │  │
  │  ├── FilterForm
  │  │  ├── __tests__
  │  │  ├── FilterForm.vue
  │  │  ├── SortInput.vue
  │  │  └──index.ts
  │  │
  │  ├── ProductItem
  │  ├── StatusLabel
  │  └── UserCard
  │
  ├── composables
  ├── constants
  ├── types
  └── helpers
```

Внутри index.ts импортируем и экспортируем наружу только необходимый компонент DocumentsTable.vue. А вот
TableCheckbox.vue не экспортируем наружу, так как он используется только в рамках DocumentsTable, таким образом он
остается изолированным, так как используется только здесь.

### Основные правила для слоя Shared:

- Слой shared это очень простые common-сущности, из которых будут строиться элементы в `routes/*`;
- То есть компоненты, менее самостоятельные сущности, и могут быть повторно переиспользованны на разных страницах и в
  разных модулях.
- Это делает их относительно глупыми по сравнению со слоями выше.
- Общие компоненты, могут содержать небольшую бизнес-логику, но как правило она или очень простая, или бизнес-логики нет
  совсем.
- Компоненты могут содержать внутри себя только ui-компоненты из слоя ниже (UI);
- Категорически запрещено: обращаться внутри слоя shared к слоям высшего порядка `routes/*`.

## Группы роутов/Модули (`routes/*`)

Ввиду особенностей domain-driven файловой системы в `Vike` складывается следующая структура - модуль верхнего уровня
это `shared` и подмодули в виде `pages/*`.

Например модуль product:

```
src/routes/product/
  │
  ├── node_modules
  ├── .eslintrc.cjs
  ├── package.json
  ├── pages
  │  │
  │  ├── +config.ts
  │  ├── +Head.vue
  │  ├── +Layout.vue
  │  ├── browse
  │  ├── brands
  │  └── search
  │    │
  │    ├── +Page.vue
  │    ├── +Layout.vue
  │    ├── +data.ts
  │    ├── +title.ts
  │    ├── __tests__
  │    ├── api
  │    ├── components
  │    ├── composables
  │    ├── constants
  │    ├── types
  │    ├── helpers
  │    └── .eslintrc.cjs
  │
  └── shared
    │
    ├── __tests__
    ├── api
    ├── components
    ├── composables
    ├── constants
    ├── types
    ├── helpers
    └── .eslintrc.cjs
```

### Основные правила для модулей `routes/*` :

- Каждый модуль это изолированная самостоятельная сущность;
- Модуль это основной источник бизнес-ценности;
- Модуль содержит внутри себя всё необходимое для работы этого конкретного модуля и реализации бизнес-логики;
- Все хэлперы, константы, апи, стор, и тд. не разбросаны по проекту, а изолированы в своих модулях;
- Модуль может быть большим, и может содержать в себе неограниченное количество внутренних компонентов, характерных для
  конкретного модуля;
- При этом внутренние компоненты модуля, могут переиспользовать компоненты из общих слоев: 1) `UI` 2) уровня выше
  слой `../shared`
- Важно: Модули не должны внутри себя использовать другие модули (это позволит избегать кольцевых зависимостей и сильной
  связанности модулей), если нам нужно объединить два модуля, то нужно делать это уже на уровне выше на слое `shared`;
- Важно: Внутренние компоненты модуля должны решать одну конкретную задачу, связанную с этим модулем;
- Важно: Импорты внутри модуля, должны использовать alias, для возможности линтинга и изоляции
