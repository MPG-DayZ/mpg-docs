# Container и Presentational компоненты

Как следует из названия, этот шаблон позволит нам отделить презентационный компонент от бизнес-логики, поощряя
разделение интересов. Презентационные компоненты могут быть чистыми функциями, которые отвечают за пользовательский
интерфейс, тогда как компоненты-контейнеры отвечают за состояние и данные приложения. Это упрощает реализацию разделения
задач.

Контейнер может извлекать, обрабатывать и изменять данные, которые будут показаны пользователю. Их обязанность —
передавать эти данные (которые они содержат) презентационному компоненту.

Презентационные компоненты легко сделать повторно используемыми, поскольку они просто отображают данные, не изменяя их.
Мы можем повторно использовать презентационные компоненты в нашем приложении для разных целей.

Поскольку презентационные компоненты не изменяют логику приложения, внешний вид презентационных компонентов может быть
легко изменен кем-то без знания кодовой базы, например дизайнером. Если презентационный компонент был повторно
использован во многих частях приложения, изменение может быть единообразным во всем приложении.

Тестирование презентационных компонентов легко, поскольку они обычно являются чистыми функциями. Мы знаем, что будут
отображать компоненты на основе переданных нами данных, без необходимости имитировать хранилище данных.

Есть полезная [статья](https://medium.com/@dan_abramov/smart-and-dumb-components-7ca2f9a7c7d0) от Дэна Абрамова, в
которой он более детально описывает нюансы и приводит примеры данного паттерна

## Пример использования

Попробуем разобрать данный паттерн на примере продуктовых полок. Учитывая специфику Vike, и представив что получить их
нужно на первом экране, и что нужно отрисовать их на сервере (но в целом это неважно, можно и на клиенте)

Получаем продукты для например бренд и рекомендационных полок:

```ts
// src/routes/home/pages/index/+data.ts
// Environment: server

export { data };
export type Data = Awaited<ReturnType<typeof data>>;

import { ofetch } from 'ofetch';

async function data(pageContext: PageContextServer) {
  const recommendProductsData = await ofetch(`/api/products/recommend`);
  const brandProductsData = await ofetch(`/api/products/brand`);

  return {
    recommendProductsData,
    brandProductsData,
  };
}
```

Далее реализуем Презентационный компонент ProductList.vue

```vue
// ProductList.vue
<script setup>
const props = defineProps({
  products: {
    type: Array,
    default: () => [],
  },
});

const emit = defineEmits(['click']);

function handleClick(index) {
  emit('click', index);
}

defineExpose({
  products: props.products,
  handleClick,
});
</script>

<template>
  <ul class="product-list">
    <li v-for="(product, i) in products">
      <Product :data="product">
        <button @click="handleClick(i)" class="product-list__button">Click</button>
      </Product>
    </li>
  </ul>
</template>
```

В чем удобство - мы определяем props и events, которые может емитить компонент, и все. Если мы передаем бренды, то будут
отображаться бренды, если мы передаем рекоммендации, то будут отображаться рекоммендации. Если в будущем летутех решит
показать еще какую-то полку, то мы застрахованы. Модульные тесты компонентов становятся проще простого, и вы можете
очень легко добавить компонент в Storybook или другие библиотеки компонентов.

А теперь создадим контейнер для BrandList.vue/RecommendList.vue

```ts
import { onMounted, h } from 'vue';
import ProductList from './ProductList.vue';
import { useData } from 'vike-vue/useData';
import type { Data } from './+data';

export default {
  setup() {
    const { brandProductsData, recommendProductsData } = useData<Data>();

    function handleOnClick(index) {
      console.log('Clicked on brandProduct/recommendProduct item #' + index);
    }

    return () =>
      h(ProductList, {
        products: brandProductsData.value / recommendProductsData.value,
        onClick: handleOnClick,
      });
  },
};
```

Этот контейнер отвечает за получение данных продуктов из +data() на сервере (точнее Vike выполнит все запросы на
получение данных описанных в +data.ts, но для понимания паттерна сделаем преднамеренно такую оговорку, либо если это
клиентский запрос - то это будет фетч данных при монтировании компонента), и далее при помощи useData() получение
конкретных требуемых данных и визуализацию компонента ProductList SFC, передавая список товаров как свойство. Он также
прослушивает события, которые может генерировать компонент ProductList, и обрабатывает их.

Это всего лишь простой пример: в данных условиях логика внутри контейнеров может существенно меняться, но важно то, что
ваша логика компонента ProductList.vue останется прежней. Компонент попрежнему использует те же props, и пока другие
контейнеры соблюдают этот интерфейс, он должен отображать то что требуется - список продуктов.
